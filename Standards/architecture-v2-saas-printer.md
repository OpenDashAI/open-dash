# OpenDash Architecture v2 — SaaS Printer

**Date**: 2026-03-25
**Status**: Definitive — supersedes all prior architecture documents

---

## Vision

OpenDash is a SaaS printer. Describe what you need, AI generates a complete micro-SaaS from structural component primitives — customized for your domain, composed via event contracts, backed by a unified data layer.

The architecture has seven layers — five already built, two to build:

```
Layer 1: DATA           Scram Jet YAML pipelines → D1 unified storage          ✅ EXISTS
Layer 2: AUTH           Clerk + RBAC + multi-tenancy + org management           ✅ EXISTS
Layer 3: TRANSPORT      HudSocket DO + WebSocket + useLiveData                  ✅ EXISTS
Layer 4: SERVER ACTIONS TanStack server functions + D1 mutations                ✅ EXISTS
Layer 5: HOOKS          Headless composition hooks (useComposableList, etc.)    🔨 TO BUILD
Layer 6: GENERATION     AI customizes hooks into domain components (once)       🔨 TO BUILD
Layer 7: COMPOSITION    AI wires domain components → JSON → runtime renders    🔨 TO BUILD
```

### What Already Exists (and doesn't need to be rebuilt)

**Auth** (src/server/auth.ts, rbac.ts, auth-middleware.ts, worker-context.ts):
- Clerk session validation, login page
- RBAC: owner/editor/viewer roles with permission matrix
- Org management: create, invite, tier limits (starter/pro/enterprise)
- Per-request auth context injection

**Real-time Transport** (src/server/hud-socket.ts, src/routes/api/ws.tsx, src/lib/use-live-data.ts):
- HudSocket Durable Object (WebSocket hub, broadcast, hibernation)
- Client hook with auto-reconnect, heartbeat, fallback polling
- Needs: bridge to CompositionProvider (emit remote events)

**Server Actions** (src/routes/api/*.ts, src/server/*.ts):
- 15+ API endpoints with auth + permissions
- TanStack Start server functions
- D1 CRUD via Drizzle ORM
- AI chat with 40+ orchestrator tools

## Layer 1: Data — Scram Jet + D1

**What**: All external data collection and internal storage.

**External data** (read from the world):
- Scram Jet YAML pipelines connect to any API (Stripe, GA4, Google Ads, etc.)
- 50+ pre-built operators, 300+ possible via OpenAPI auto-generation
- Data lands in D1 `metrics` table (source, category, priority, title, detail, metadata)
- Adding a new data source = writing a YAML file (minutes)

**Internal data** (SaaS application state):
- D1 tables for domain data (contacts, invoices, timers, tickets, etc.)
- Schema generated by AI at print time based on the domain
- Standard CRUD operations via server functions

**Key principle**: Components never call external APIs. They read/write D1. Scram Jet handles all external communication.

### Files
- `workers/metrics-collector/` — Scram Jet → D1 bridge
- `src/lib/db/schema.ts` — D1 schema (Drizzle ORM)
- Scram Jet pipelines live in the scram-jet repo (garywu/scram-jet)

## Layer 2: Primitives — Headless Composition Hooks

**What**: Headless hooks that add event-driven wiring to ANY styled component. Zero styling, zero UI opinions. Any design system (shadcn, Magic UI, custom) can use these hooks.

```
Composition hook (headless — events only)     ← what we build
Design system (shadcn / Magic UI / custom)    ← chosen at print time
Domain component (composition + styling)      ← AI generates this
```

Each hook:
- **Is purely headless** (returns data + emit functions, NEVER JSX)
- **Registers** with CompositionProvider on mount
- **Emits events** on interaction (item-selected, form-submitted, etc.)
- **Listens to events** from other components (data-ready, filter-updated, etc.)
- Has **customization slots** (data shape, fields, event payloads)
- Is described in the registry (`public/r/index.json`) with event contracts

**The differentiation is NOT the UI components (shadcn has those). It's the headless composition layer that makes ANY component composable by AI.**

### The Primitive Library

**Display primitives** (~6):
| Primitive | Structure | Customizable |
|-----------|-----------|-------------|
| `List` | Scrollable items, selection, filtering | Item renderer, data shape, empty state |
| `Table` | Rows + columns, sorting, pagination | Column definitions, row actions |
| `KanbanBoard` | Columns with draggable cards | Column names, card renderer, statuses |
| `Calendar` | Month/week/day views, event slots | Event renderer, date range, interactions |
| `Timeline` | Chronological entries | Entry renderer, grouping, time format |
| `Card` | Single content container with header/body/footer | Content slots, actions, styling variant |

**Input primitives** (~5):
| Primitive | Structure | Customizable |
|-----------|-----------|-------------|
| `Form` | Labeled fields, validation, submit | Field definitions, validation rules, layout |
| `Search` | Text input with debounce, emits query | Placeholder, filters, result format |
| `Filter` | Multi-criteria filtering panel | Filter fields, operators, defaults |
| `DatePicker` | Date/range selection | Format, range constraints, presets |
| `FileUpload` | Drag-drop + browse, preview | Accept types, size limits, preview renderer |

**Action primitives** (~3):
| Primitive | Structure | Customizable |
|-----------|-----------|-------------|
| `ActionButton` | Button that emits an event with confirmation | Label, icon, confirmation dialog, payload |
| `ApprovalFlow` | Approve/reject with reason | Steps, required fields, recipients |
| `ActionMenu` | Dropdown of context-specific actions | Menu items, icons, keyboard shortcuts |

**Visualization primitives** (~3):
| Primitive | Structure | Customizable |
|-----------|-----------|-------------|
| `Chart` | Line/bar/pie with axes | Series definitions, colors, axis labels |
| `MetricCard` | Big number + trend + label | Value format, comparison, icon |
| `ProgressBar` | Progress toward goal | Label, thresholds, color stops |

**Time primitives** (~2):
| Primitive | Structure | Customizable |
|-----------|-----------|-------------|
| `Timer` | Countdown/countup with controls | Duration, format, linked behavior |
| `Scheduler` | Scheduled items on a timeline | Item renderer, recurrence, constraints |

**Communication primitives** (~2):
| Primitive | Structure | Customizable |
|-----------|-----------|-------------|
| `MessagePanel` | Send/receive messages | Message format, recipients, channels |
| `NotificationBar` | Alert banner with dismiss/action | Severity levels, auto-dismiss, actions |

**Total: ~21 structural primitives**

### Key Principle

Primitives are **structural**, not domain-specific. A `List` is not a `ContactList` or a `TimerList` — it's the structure that both are generated from.

### Files
- `src/components/primitives/` — Structural primitives (to be built)
- `public/r/index.json` — Registry with customization slot metadata

## Layer 3: Generation — AI Customizes Primitives

**What**: AI takes structural primitives and generates domain-specific components by filling in customization slots. This happens ONCE at print time.

### How It Works

```
Input:  User describes their SaaS ("I need a CRM")
        AI understands the domain (contacts, deals, activities)

Step 1: AI selects relevant primitives
        List → for contacts
        KanbanBoard → for deal pipeline
        Timeline → for activity history
        Form → for adding contacts/deals
        Chart → for pipeline analytics

Step 2: AI customizes each primitive (GENERATES domain component)

        List + CRM context =
          ContactList {
            dataShape: { name: string, email: string, company: string, status: string }
            itemRenderer: name bold, email gray, company tag, status badge
            emits: "contact-selected" with Contact payload
            styling: compact rows, avatar, hover highlight
          }

        KanbanBoard + CRM context =
          DealPipeline {
            columns: ["Lead", "Qualified", "Proposal", "Negotiation", "Closed"]
            cardRenderer: deal name, value, company, days in stage
            emits: "deal-moved" with { deal, fromStage, toStage }
            styling: colored columns, drag handles
          }

Step 3: AI writes these as real files (shadcn model)
        → src/components/domain/ContactList.tsx
        → src/components/domain/DealPipeline.tsx
        → src/components/domain/ActivityTimeline.tsx
        etc.

Step 4: Components are added to the project's registry
        with full event contracts for Layer 4
```

### What AI Decides (Once, at Generation Time)
- **Data shape**: What fields each component works with
- **Event payloads**: What each component emits/listens to
- **Styling**: Visual presentation for this domain
- **Layout**: Where components sit (sidebar, main, panel)
- **Labels**: Domain-specific text (Contact vs. Timer vs. Ticket)
- **Validation**: Business rules for forms
- **Actions**: What write operations are available

### What AI Does NOT Decide
- **Structure**: Comes from the primitive (List is always a scrollable item container)
- **Event system**: Always CompositionProvider (emitEvent/onComponentEvent)
- **Data layer**: Always D1 (read/write via server functions)

### Output
Generated domain components are concrete files in the project. You own them. You can modify them. But they were generated from primitives + AI customization, not written from scratch.

## Layer 4: Composition — AI Wires Components

**What**: AI connects generated domain components into a working application by wiring event chains. This is the same constraint satisfaction from the v1 architecture.

### How It Works

```
Generated components (from Layer 3):
  ContactList     emits: contact-selected
  DealPipeline    emits: deal-moved     listens: contact-selected
  ActivityTimeline                      listens: contact-selected
  ContactForm     emits: contact-created
  PipelineChart                         listens: deal-moved

AI composes:
<CompositionProvider>
  <Sidebar>
    <ContactList componentId="contacts" />
    <ContactForm componentId="new-contact" />
  </Sidebar>
  <Main>
    <DealPipeline componentId="deals" listenToComponent="contacts" />
    <ActivityTimeline componentId="activity" listenToComponent="contacts" />
  </Main>
  <Panel>
    <PipelineChart componentId="analytics" listenToComponent="deals" />
  </Panel>
</CompositionProvider>

Validates:
  contact-selected → DealPipeline ✓ (matches)
  contact-selected → ActivityTimeline ✓ (matches)
  deal-moved → PipelineChart ✓ (matches)
  No dangling listeners ✓
  No cycles ✓
```

### Composition Format

AI outputs a declarative composition (JSON) that the runtime renders:

```json
{
  "name": "Simple CRM",
  "layout": "sidebar-main-panel",
  "components": [
    { "id": "contacts", "component": "ContactList", "slot": "sidebar" },
    { "id": "new-contact", "component": "ContactForm", "slot": "sidebar" },
    { "id": "deals", "component": "DealPipeline", "slot": "main", "listenTo": "contacts" },
    { "id": "activity", "component": "ActivityTimeline", "slot": "main", "listenTo": "contacts" },
    { "id": "analytics", "component": "PipelineChart", "slot": "panel", "listenTo": "deals" }
  ]
}
```

Runtime reads this and renders the CompositionProvider tree. No manual JSX needed.

## Shared Contract: @opendash/composition

The `@opendash/composition` npm package is the ONE shared dependency:
- `CompositionProvider` — event bus
- `CompositionContext` — interface (registerComponent, emitEvent, onComponentEvent)
- `useCompositionContext` — React hook

Any application in the ecosystem imports this package. Everything else (primitives, domain components, compositions) is local to each project.

### Files
- `packages/@opendash/composition/` — The shared package
- `docs/COMPONENT_MESH_ARCHITECTURE.md` — How to build composable components

## Full Flow: Printing a SaaS

```
1. USER: "I need a stage timer for live events"

2. LAYER 1 (Data):
   AI generates D1 schema: timers(id, label, duration, status, order)
   No external APIs needed (or Scram Jet if integrating with calendar, etc.)

3. LAYER 2 (Primitives):
   AI selects: List, Timer, ActionButton, MessagePanel, Card

4. LAYER 3 (Generation):
   AI generates domain components:
   - TimerList (List + timer context)
   - TimerDisplay (Timer + fullscreen countdown context)
   - TimerControls (ActionButton + play/pause/skip context)
   - MessageOverlay (MessagePanel + stage message context)
   - FlashAlert (Card + attention-grabber context)

5. LAYER 4 (Composition):
   Controller: TimerList → TimerControls → MessageOverlay
   Viewer: TimerDisplay + MessageOverlay + FlashAlert
   AI generates composition JSON for both views

6. OUTPUT: Working stage timer app
   - 5 generated components (from primitives)
   - 2 compositions (controller + viewer)
   - D1 schema for timer data
   - Total generation time: minutes
```

## Principles

1. **Primitives define structure, AI defines opinion.** Never bake domain logic into primitives.
2. **Generate once, own forever.** AI creates domain components at print time. You own the files (shadcn model).
3. **Events are the contract.** Components communicate through CompositionProvider, nothing else.
4. **D1 is the data layer.** External data via Scram Jet, internal data via D1. Components only talk to D1.
5. **Registry enables AI.** Primitives and domain components are described in the registry so AI can discover, generate, and compose.
6. **Each print grows the registry.** Domain components generated for one SaaS become available for future prints.
