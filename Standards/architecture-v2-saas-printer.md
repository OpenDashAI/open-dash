# OpenDash Architecture v2 — SaaS Printer

**Date**: 2026-03-25
**Status**: Definitive — supersedes all prior architecture documents

---

## Vision

OpenDash is a SaaS printer. Describe what you need, AI generates a complete micro-SaaS — customized for your domain, composed via validated event contracts, styled from proven real-world designs, backed by production infrastructure.

100% of SaaS front-ends can use this architecture. The developer's only job is the business logic inside each component.

---

## The 9-Layer Stack

```
Layer 1: DATA              Scram Jet YAML pipelines → D1                    ✅ EXISTS
Layer 2: AUTH              Clerk + RBAC + multi-tenancy + orgs              ✅ EXISTS
Layer 3: TRANSPORT         HudSocket DO + WebSocket + useLiveData           ✅ EXISTS
Layer 4: SERVER ACTIONS    TanStack server functions + D1 mutations         ✅ EXISTS
Layer 5: HOOKS             Headless composition hooks                       ✅ BUILT
Layer 6: COMPOSITION       JSON format → CompositionRenderer                ✅ BUILT
Layer 7: GENERATION        AI: hook + domain context → domain component    ✅ PROVEN
Layer 8: DESIGN RETRIEVAL  Real app designs → composition JSON database     🔨 NEXT
Layer 9: AI ORCHESTRATOR   User intent → retrieve design → generate → compose  🔨 NEXT
```

## Layer 1–4: Infrastructure (Already Built)

### Layer 1: Data — Scram Jet + D1
- YAML pipelines collect ALL external data (Stripe, GA4, Google Ads, etc.)
- D1 unified metrics table (source, category, priority, title, detail, metadata)
- Components NEVER call external APIs — they read/write D1
- Internal state (contacts, invoices, timers) also stored in D1
- Schema generated by AI at print time based on domain

### Layer 2: Auth — Clerk + RBAC
- Clerk session validation, login page
- RBAC: owner/editor/viewer with permission matrix
- Org management: create, invite, tier limits (starter/pro/enterprise)
- Per-request auth context injection

### Layer 3: Transport — HudSocket DO
- Durable Object WebSocket hub (broadcast, hibernation)
- Client hook with auto-reconnect, heartbeat, fallback polling
- Enables multi-device apps (controller ↔ viewer)
- Needs: bridge to CompositionProvider (#133)

### Layer 4: Server Actions — TanStack Start
- 15+ API endpoints with auth + permissions
- D1 CRUD via Drizzle ORM
- AI chat with 40+ orchestrator tools
- Enables write operations from composition events (#134)

## Layer 5–7: Composition System (Built + Proven)

### Layer 5: Hooks — Headless Composition Primitives
5 headless hooks that add event-driven wiring to ANY styled component:

| Hook | Behavior | Events |
|------|----------|--------|
| `useComposableList` | Items, selection, add/remove, filtering | item-selected, item-added, item-removed |
| `useComposableForm` | Fields, validation, submit, pre-fill | form-submitted, form-changed |
| `useComposableCard` | Detail view, data from selection, actions | card-action |
| `useComposableChart` | Series data, point selection, range | point-selected, range-changed |
| `useComposableTimer` | Countdown/countup, start/stop/reset | timer-started, timer-stopped, timer-complete |

**Key**: Hooks return data + emit functions. NEVER JSX. Any design system (shadcn, Magic UI, custom) can consume them.

**Shared contract**: `@opendash/composition` package (CompositionProvider, event bus)

### Layer 6: Composition — JSON → Runtime
- Zod schema for declarative compositions
- `CompositionRenderer` reads JSON + component map → renders full app
- Layouts: single, sidebar-main, sidebar-main-panel, header-main, grid
- Validates event wiring (dangling listeners, missing components)
- Slot-based layout with CSS class overrides

### Layer 7: Generation — AI Fills In Opinions
AI takes a headless hook + domain context and generates a domain-specific component:

```
useComposableList + "CRM contacts" → ContactList.tsx
useComposableList + "stage timer schedule" → TimerList.tsx
useComposableList + "booking calendar" → DatePicker.tsx
```

Same hook, different domain = different component. Generated once, owned forever (shadcn model).

**Proven with 4 apps:**

| App | Domain | Components Generated | Hooks Used |
|-----|--------|---------------------|------------|
| Dashboard | Analytics | DataSelector, Filter, Display, Summary | List |
| Mini CRM | CRM | ContactList, ContactDetail | List, Card |
| Stage Timer | Events | TimerList, StageTimerDisplay | List, Timer |
| Booking Page | Scheduling | EventInfoCard, DatePicker, TimeSlotPicker, BookingForm | Card, List, List, Form |

## Layer 8: Design Retrieval (Next)

### The Problem
The composition system gives you correct wiring. It doesn't give you good design. Layout proportions, visual hierarchy, transitions, empty states, responsive behavior — that's still art.

### The Solution: Design as Data
Extract real app designs into composition JSON. Build a retrieval database of thousands of proven layouts. AI retrieves the best matching design instead of generating from scratch.

### How It Works

```
1. EXTRACT: Analyze real app (screenshot or source code)
   → Vision AI extracts: layout, components, slots, styling, flow, design tokens
   → Output: composition JSON with rich metadata

2. STORE: Database of tagged composition templates
   → Tagged by: domain, layout pattern, complexity, component types
   → Includes: slot proportions, styling tokens, flow description

3. RETRIEVE: User describes what they need
   → AI queries database for best matching templates
   → Returns top N compositions ranked by relevance

4. CUSTOMIZE: AI fills template with domain components
   → Template provides layout + proportions + design tokens
   → AI generates domain components to fill the slots
   → Result: well-designed app from proven layout
```

### Design Extraction Format
Each extracted design includes:
- **Layout**: slot positions, proportions, responsive breakpoints
- **Components**: what's in each slot, which hook each uses
- **Data shapes**: TypeScript interfaces for each component's data
- **Styling**: theme (light/dark), spacing (compact/spacious), corners, typography
- **Flow**: user journey description (click this → that happens)
- **Design tokens**: colors, fonts, spacing scale

### Sources for Extraction

| Source | Volume | Quality | Method |
|--------|--------|---------|--------|
| Open source apps (cal.com, twenty, midday) | ~100 | Very high | Source code analysis |
| shadcn blocks + Tailwind UI | ~500 | Very high | Code extraction |
| SaaSFrame.io, Screenlane.com | ~15K | High | Screenshot + vision AI |
| Dribbble, Mobbin | ~100K+ | Medium | Screenshot + vision AI |
| Figma Community | ~50K | High | Design file parsing |

### Eval Loop
- Deploy compositions to users
- Measure: engagement, task completion, retention
- Feed metrics back into template rankings
- Good designs rise, bad designs sink
- Database self-curates over time

## Layer 9: AI Orchestrator (Future)

The end-to-end flow when a user says "I need a booking tool":

```
1. UNDERSTAND intent
   → "booking tool" → domain: scheduling, components: calendar, time slots, form

2. RETRIEVE design
   → Query design database: domain=booking
   → Returns: cal.com-style layout (sidebar-main, event info + calendar + slots + form)

3. GENERATE schema
   → D1 tables: bookings, events, time_slots, users
   → Drizzle ORM + server functions

4. GENERATE components
   → useComposableList + booking domain → DatePicker, TimeSlotPicker
   → useComposableForm + booking domain → BookingForm
   → useComposableCard + booking domain → EventInfoCard

5. COMPOSE
   → Wire components via event contracts
   → Validate: every listener has matching emitter
   → Output: composition JSON

6. RENDER
   → CompositionRenderer reads JSON → working app
   → Auth (Clerk), data (D1), transport (HudSocket) all already there

Total: user describes → working production app
```

## Competitive Differentiation vs. Vibe-Coding Tools

### v0 / Lovable / Bolt / Replit Agent

These tools generate ALL code from scratch every time:
- Prompt → LLM → 2,000 lines of code → hope it works
- No reuse between projects
- No validated wiring
- No production infrastructure (auth, DB, real-time)
- Quality is probabilistic

### OpenDash SaaS Printer

We decompose into layers with different guarantees:

| Layer | Guarantee | v0/Lovable |
|-------|-----------|-----------|
| Hooks | Tested code, guaranteed correct | Generated, hope it works |
| Wiring | Mechanically validated event contracts | Generated, hope it connects |
| Design | Retrieved from proven real apps | Generated, hope it looks good |
| Auth/DB/Realtime | Already built, production-ready | Generated stubs each time |
| Domain logic | AI-generated (the only uncertain part) | AI-generated (everything uncertain) |

### The Key Insight

**v0 generates prototypes. OpenDash generates production systems.**

- v0: generates code → throw it away → generate more code
- OpenDash: retrieves proven design → fills with validated composition → deploys on production infra

For quick prototypes: v0 is faster (30 seconds).
For real apps: OpenDash is better (production-ready, composable, evolvable).

### Where Our Advantage Compounds

| After N apps | v0/Lovable | OpenDash |
|-------------|-----------|---------|
| 1 app | Fast generation | Same speed |
| 5 apps | 5x generation cost | Hooks reused, design DB growing |
| 50 apps | 50x generation cost | Most components exist, near-zero cost per new app |

Each print grows the registry and design database. Their cost is linear. Ours compounds.

## The Business Thesis

100% of SaaS front-ends can use the composition system. The wiring, structure, auth, data, transport, UI — all of it. The developer's only job is the business logic inside each component.

For simple products (timer, dashboard, booking): ~90% done by the system. Minimal business logic.
For complex products (accounting, healthcare): System handles 100% of front-end wiring. Developer writes substantial business logic inside components.

Each print adds to the registry. The registry is the moat. The more prints, the cheaper the next print.

## Principles

1. **Hooks define structure, AI defines opinion.** Never bake domain logic into hooks.
2. **Generate once, own forever.** AI creates domain components at print time (shadcn model).
3. **Events are the contract.** Components communicate through CompositionProvider, nothing else.
4. **D1 is the data layer.** External data via Scram Jet, internal data via D1.
5. **Retrieve design, don't generate it.** Proven layouts from real apps beat AI-generated layouts.
6. **Each print grows the registry.** Components and designs compound.
7. **100% front-end, developer adds business logic.** The system handles everything except domain-specific rules.
